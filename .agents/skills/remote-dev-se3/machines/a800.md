## A800 源码同步 fallback：git bundle

如果 A800 pod 里 `git fetch` / `git pull` 访问 GitHub 报：

```text
gnutls_handshake() failed: Error in the pull function
```

不要继续硬拉 GitHub。改用本地生成 git bundle，再传到宿主机和 pod 内执行 `git fetch <bundle> HEAD`。这仍然是 git 同步方式，比 rsync 覆盖源码可追溯。

本地 PowerShell：

```powershell
$bundle = "recovery_$(git rev-parse --short HEAD).bundle"
git bundle create $bundle HEAD
scp -P 2222 $bundle root@192.168.43.231:/tmp/$bundle
ssh root@192.168.43.231 -p 2222 "kubectl cp /tmp/$bundle llm-69445bf49f-7qq8g:/tmp/$bundle"
```

pod 内：

```bash
cd /workspace/3SE-Competitive-Robotics-Team/se3_wheel_leg
git fetch /tmp/recovery_805620d.bundle HEAD
git switch <branch>
git merge --ff-only FETCH_HEAD
git rev-parse --short HEAD
```

如果远端还没有对应分支，用 `git switch -c <branch> FETCH_HEAD`。如果 `merge --ff-only` 失败，说明远端分支有本地提交或脏状态；先查 `git status` 和 `git log --oneline --decorate -5`，不要用 rsync 覆盖源码。

非交互 `kubectl exec ... bash -lc` 不保证加载交互 shell 的 PATH，pod 里 `uv` 常见报错是：

规则：
- 必须用 uv 时写绝对路径 `/root/.local/bin/uv`，或先 `export PATH=/root/.local/bin:$PATH`。
- 训练命令若需要 uv 入口，优先加 `--no-sync`，避免在训练/回放途中隐式联网更新环境。
- 回放、Rerun 录制和离线评估优先用 `.venv/bin/se3-sim2sim`、`.venv/bin/python` 等已安装入口，避免 `uv run` 因缺包去下载。

例如 pitch 轴反转回放：

  --checkpoint <checkpoint> \

如果 `.venv/bin/se3-sim2sim` 报缺包，不要在录制脚本里临时 `uv run`；先在维护窗口用代理修复环境，再重新录制。

## 产物拉取 warning 判定

`kubectl cp` 内部依赖 tar。脚本同时尝试拉 `.done` 和 `.failed` marker 时，缺少其中一个会出现类似：

```text
tar: .../model_2999.failed: Cannot stat: No such file or directory
```

这类 warning 通常是无害的：同一个 checkpoint 只会有 `.done` 或 `.failed` 之一。判断拉取是否成功时看三件事：

1. `.rrd`、summary JSON、checkpoint 等主产物是否存在且非空。
2. `.processed/model_<iter>.done` 或 `.processed/model_<iter>.failed` 是否至少存在一个。
3. 必需文件的 `kubectl cp` / `scp` 是否返回非零；可选 marker 的 `Cannot stat` 不单独判失败。
